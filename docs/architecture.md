# Architecture

This document gives a maintainer-level overview of how Codex Switcher is put together.

## High-level structure

Codex Switcher is a Wails v2 application. The Go backend owns the desktop window, tray behavior, file and process actions, and the LAN/browser runtime. The React frontend renders the GUI and calls backend methods through the Wails bridge.

```text
main.go
  -> starts the Wails app
  -> loads the frontend bundle
  -> wires in the App backend

app.go
  -> backend methods exposed to the UI
  -> account operations, backups, usage, LAN mode, window helpers

tray.go
  -> system tray actions and menu handling

src/
  -> React UI, shadcn-style components, hooks, and shared utilities

src/wailsjs/
  -> generated Wails bindings

src-electron/
  -> browser/LAN mode entry points and server runtime
```

## Frontend

The frontend lives in `src/` and is a standard React + TypeScript app.

Important areas:

- `src/App.tsx` - top-level layout and screen orchestration
- `src/components/` - reusable UI pieces and dialogs
- `src/components/ui/` - shadcn-style primitives
- `src/hooks/` - account and runtime hooks
- `src/lib/` - platform helpers and utility functions
- `src/types/` - shared TypeScript types

The frontend talks to the backend through helpers in `src/lib/platform.ts`. That layer hides the differences between Wails, browser/LAN mode, and any other supported runtime.

## Backend

The Go side is responsible for the stateful parts of the app:

- starting and configuring the window
- exposing backend methods to the frontend
- reading and writing local account state
- handling tray and window actions
- serving the browser/LAN runtime
- interacting with Codex account files and processes

The main entry point is `main.go`, which bootstraps Wails and passes the `App` type to the runtime. The primary backend logic lives in `app.go`, while `tray.go` handles tray-specific behavior.

## Browser / LAN mode

Codex Switcher can run the same UI over HTTP for browser access. That mode is built from the code in `src-electron/`.

The typical flow is:

1. Build the frontend bundle.
2. Start the LAN server entry point.
3. Serve the same UI and backend actions over HTTP.

This is useful for local browser access and controlled LAN use. It is not a public remote deployment target.

## Data flow

The UI is the entry point for user actions, but the backend owns the actual account changes.

```text
User clicks a button in the GUI
  -> React calls a helper in src/lib/platform.ts
  -> The helper invokes a Wails backend method
  -> Go reads or updates local state
  -> Go returns a typed result to the UI
  -> The UI updates the visible state
```

## Generated files

`src/wailsjs/` is generated by Wails. Do not edit those files manually.

## Build output

The standard desktop build ends up in `build/bin/`.

The Windows single-file launcher build used for distribution is created by the custom build script in `scripts/build-windows.mjs`.
